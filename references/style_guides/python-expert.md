# Python Expert Newsletter Style Guide

This style guide is for the "Pythonic" expert who values readability, performance, and clean design.

## Tone & Voice
- **Tone**: Pragmatic, expert-level, and "Pythonic" (Zen of Python).
- **Voice**: "Simple is better than complex. Complex is better than complicated."
- **Precision**: Use exact technical terms (e.g., "dunder methods", "comprehensions", "decorators").
- **Efficiency**: Focus on "the one obvious way" to do things.

## Structural Requirements
- **Title**: Clean and descriptive (`# Python Patterns: Episode XX`).
- **Idiomatic Examples**: Every episode MUST include a code snippet or pseudo-code that demonstrates the "right" way to implement the concept.
- **Deep Dives**: Explain *why* a particular design pattern or implementation is better for performance or readability.
- **Visuals**: Use Mermaid diagrams for class hierarchies, data flows, or execution cycles.

## Formatting Standards
- **PEP 8 Compliance**: All code examples MUST follow PEP 8.
- **Standard Library First**: Prioritize built-in solutions before suggesting external libraries.
- **Docstring Style**: Use callouts like `> **Pro-Tip**: ...` for advanced optimizations.
- **Clear Tables**: Use tables for performance benchmarks or API comparisons.

## Prohibitions
- **No Boilerplate**: Avoid unnecessary setup code. Get straight to the logic.
- **No Non-Idiomatic Patterns**: Do not use "C-style" patterns in Python (e.g., manual loop indexing).
- **No Obscure Hacks**: Favor readability over "clever" one-liners that are hard to maintain.
